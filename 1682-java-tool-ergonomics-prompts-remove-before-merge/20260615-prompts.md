### Schema generation from record components

```java
package com.github.copilot.rpc;

import java.lang.reflect.RecordComponent;
import java.util.*;

final class SchemaGenerator {

    static Map<String, Object> fromRecord(Class<? extends Record> type) {
        RecordComponent[] components = type.getRecordComponents();
        Map<String, Object> properties = new LinkedHashMap<>();
        List<String> required = new ArrayList<>();

        for (RecordComponent comp : components) {
            Map<String, Object> prop = new LinkedHashMap<>();
            prop.put("type", jsonType(comp.getType()));

            Param param = comp.getAnnotation(Param.class);
            if (param != null) {
                prop.put("description", param.value());
                if (param.required()) {
                    required.add(comp.getName());
                }
            } else {
                // No annotation → required by default
                required.add(comp.getName());
            }

            properties.put(comp.getName(), prop);
        }

        Map<String, Object> schema = new LinkedHashMap<>();
        schema.put("type", "object");
        schema.put("properties", properties);
        if (!required.isEmpty()) {
            schema.put("required", required);
        }
        return schema;
    }

    private static String jsonType(Class<?> type) {
        if (type == String.class) return "string";
        if (type == int.class || type == Integer.class
                || type == long.class || type == Long.class) return "integer";
        if (type == double.class || type == Double.class
                || type == float.class || type == Float.class) return "number";
        if (type == boolean.class || type == Boolean.class) return "boolean";
        if (type.isArray() || List.class.isAssignableFrom(type)) return "array";
        return "object";
    }
}
```

---

### LangChain4j: **Annotation-on-method** approach

```java
@Tool("Get current weather for a location")
String getWeather(@P("City name") String location,
                  @P(value = "Unit", required = false) String unit) {
    return fetchWeather(location, unit);
}
```

The framework discovers tools by scanning an **object's methods** for `@Tool`. It then:
1. Reflects on the `Method` to get the name (from `@Tool(name=...)` or the method name itself)
2. Reflects on each `Parameter` to build the JSON Schema (using `@P` for description/required)
3. Generates a `ToolSpecification` (name + description + schema) automatically
4. At invocation time, dispatches to the annotated method with deserialized args passed as method parameters

**The user never writes a schema or a handler lambda** — the method IS the handler.

---

### My record-based proposal: **Record-as-schema + explicit lambda** approach

```java
record WeatherArgs(@Param("City name") String location,
                   @Param(value = "Unit", required = false) String unit) {}

var tool = ToolDefinition.define("get_weather", "Get current weather",
        WeatherArgs.class,
        (args, invocation) -> CompletableFuture.completedFuture(fetchWeather(args.location(), args.unit())));
```

The record defines the schema shape; the lambda is the handler.

---

### Key differences

| Aspect | LangChain4j (`@Tool` on method) | Record-based proposal |
|--------|--------------------------------|----------------------|
| **Discovery** | Automatic — scan object for `@Tool` methods | Manual — you explicitly call `define(...)` |
| **Tool name** | Derived from method name or `@Tool(name=...)` | Provided as a string argument |
| **Description** | `@Tool("description")` on the method | Provided as a string argument |
| **Parameter schema** | Generated from method parameter types + `@P` annotations | Generated from record components + `@Param` annotations |
| **Handler** | The annotated method itself — no lambda | Explicit `TypedToolHandler<T>` lambda |
| **Arg delivery** | Each parameter is a separate method argument | Single deserialized record object |
| **Return type** | Any (String sent as-is, others serialized to JSON) | Always `CompletableFuture<String>` |
| **Registration** | Implicit (pass object to framework, it finds all `@Tool` methods) | Explicit (build a `List<ToolDefinition>`) |
| **Async** | Synchronous method (framework handles threading) | Explicitly async via `CompletableFuture` |

### The fundamental philosophical difference

LangChain4j treats **the method as the tool**. You annotate existing methods and the framework does everything else. This is closest to C#'s `CopilotTool.DefineTool(SetCurrentPhase)`.

Yes — this is completely solvable with an annotation processor.

The key insight: **`javax.lang.model.element.VariableElement.getSimpleName()` always returns the real parameter name at compile time**, regardless of whether `-parameters` is passed to `javac`. The `-parameters` flag only controls whether those names survive into the `.class` bytecode for runtime reflection. An annotation processor sees the source-level names unconditionally.

### How it would work

The processor runs at compile time, finds all `@CopilotTool`-annotated methods, and generates a metadata class (or resource) that bakes in the parameter names, types, and descriptions:

```java
// GENERATED — do not edit
final class MyTools$$CopilotToolMeta {
    static List<ToolDefinition> definitions(MyTools instance) {
        return List.of(
            new ToolDefinition("set_current_phase",
                "Sets the current phase of the agent.",
                Map.of("type", "object",
                       "properties", Map.of("phase", Map.of("type", "string", "description", "The phase to transition to")),
                       "required", List.of("phase")),
                invocation -> {
                    Phase phase = invocation.getArgumentsAs(Phase.class);
                    return CompletableFuture.completedFuture(
                        instance.setCurrentPhase(phase));
                }, null, null, null)
        );
    }
}
```

At runtime, `ToolDefinition.fromObject(myTools)` just loads the generated `$$CopilotToolMeta` class — zero reflection, zero dependency on `-parameters`.

### Precedent in the ecosystem

| Framework | Approach |
|-----------|----------|
| **Micronaut** | Annotation processor generates all DI metadata at compile time — no runtime reflection, no `-parameters` needed |
| **Dagger 2** | Processor generates `_Factory` / `_MembersInjector` classes |
| **MapStruct** | Processor generates mapper implementations from interface method signatures |
| **Our own `@CopilotExperimental`** | Processor walks declared elements via JSR 269 |

### Advantages over the `-parameters` fallback

1. **No build configuration requirement** — works out of the box with any `javac` invocation.
2. **No runtime reflection** — the generated code calls the method directly. Faster, GraalVM-native-friendly, no `setAccessible` issues.
3. **Compile-time validation** — the processor can emit errors if `@Param` is missing on a parameter, if a type isn't serializable, if two tools have the same name, etc. (same as we did for `@CopilotExperimental`).
4. **Schema correctness guaranteed** — the schema is computed once from the source AST, not guessed at runtime from erased generics.

### Trade-off vs. pure runtime reflection (langchain4j default)

| | Annotation processor | Runtime reflection |
|---|---|---|
| Requires `-parameters`? | No | Yes (or `@P(name=...)`) |
| GraalVM native-image friendly? | Yes | Needs reflection config |
| Compile-time error checking? | Yes | Fails at runtime |
| Extra generated source files? | Yes | None |
| Works without running the processor? | No — but fails loudly at compile time | Yes (degraded) |

So yes: we can follow the same JSR 269 pattern as `@CopilotExperimental` to eliminate the `-parameters` requirement entirely. The processor sees the real names from source, bakes them into generated code, and the runtime never needs to call `Parameter.getName()`.
