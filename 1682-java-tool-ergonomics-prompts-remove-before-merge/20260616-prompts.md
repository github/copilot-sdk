var session = client.createSession(new SessionConfig()
        .setOnPermissionRequest(PermissionHandler.APPROVE_ALL)
        .setSystemMessage(systemMessageConfig)
        .setAvailableTools(new ToolSet().addCustom("*").addBuiltIn("web_fetch"))

                ToolDefinition.create("set_current_phase", "Sets the current phase of the agent. Use this to report progress.",

-----

The key differences from the C# version:

| C# | Java |
|---|---|
| `CopilotTool.DefineTool(SetCurrentPhase)` — infers name, description, and schema from the method's attributes via reflection | `ToolDefinition.create(name, description, schema, handler)` — all explicit |
| `new() { OverridesBuiltInTool = true }` (options object) | `ToolDefinition.createOverride(...)` (dedicated factory) |
| `[DisplayName]` / `[Description]` attributes on the method | No annotation-based discovery; you provide the strings directly |
| Implicit JSON Schema generation from parameter types | You supply the `Map<String, Object>` schema manually |

The Java SDK does not currently have a reflection/annotation-based `DefineTool` that can inspect a method and auto-generate the name, description, and parameter schema. Everything is explicit.

### 19:56 Response

 Based on the documentation, here's how skills work:

  You don't need to explicitly invoke skills. Skills are automatically loaded based on their description field in the frontmatter. The agent sees the skill listed in <available_skills> and invokes it when the task matches the description.

  However, you can be explicit if you want. Any of these would work:

   1. Implicit (recommended): Just describe what you want — the skill's description matches "creating a new Java E2E integration test that requires a new replay proxy YAML snapshot":
   Create a new failsafe IT test with a YAML snapshot that exercises the new feature.
   2. Explicit mention: Reference the skill by name to make sure it's invoked:
   Use the new-java-e2e-test-yaml-and-test skill to create a test and YAML snapshot for the new feature.
   3. Check available skills: Use /skills in the CLI to see what's loaded, or /env to see the full environment including skills.

  Both approaches work. The agent sees the skill in its available tools and will call it before generating code. The explicit mention is a good safety net if you want to be sure.
