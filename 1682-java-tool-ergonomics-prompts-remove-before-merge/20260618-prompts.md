 ●  ToolDefer  is the per-tool loading policy in  ToolDefinition —it tells Copilot whether to preload a tool  definition or allow lazy loading via tool search.

   In this repo it has exactly two values:

   1.  AUTO  ( "auto" ): tool may be deferred and surfaced through tool search.  
   2.  NEVER  ( "never" ): tool is always pre-loaded.

### 19:28 Response

 ● Short answer: you’re not blocked by the lower-level schema.

   The wire-level tool shape treats  parameters  as an open JSON object, not a closed schema with fixed keys:

   • Node RPC type:  Tool.parameters?: { [k: string]: unknown }  ( nodejs/src/generated/rpc.ts )
   • Java generated RPC type:  @JsonProperty("parameters") Map<String, Object>     ( java/src/generated/java/com/github/copilot/generated/rpc/Tool.java )
   • Go type:  Parameters map[string]any  ( go/types.go )

   So JSON Schema keywords like  default  can flow through. There is just no dedicated first-class  defaultValue   field in  ToolDefinition ; it would have to be emitted into the JSON schema for that parameter  ( properties.<name>.default ).

   Practically: adding  @Param(defaultValue=...)  is compatible at the transport layer, but likely documentation/hint  only unless you also implement SDK/runtime behavior to apply defaults when args are missing.
   
### 19:29 Prompt

Well, of course, if the user specifies a default, then we need to make it so that default does flow through to the tool invocation. Is that implementable?  I don't see why not.

### 19:31 Response

 ● Yes—it’s implementable, and you should do it in the Java SDK layer.

   The CLI/tool wire contract already passes raw argument JSON through; Java can apply defaults before invoking the  method. A solid design is:

   1. Emit  default  into the generated JSON Schema ( properties.<param>.default ) for model guidance.
   2. In generated/reflection invocation code, merge defaults into the incoming args map only when a key is absent     (not when present with explicit  null , unless you define that behavior).
   3. Then run normal coercion/casting and method invocation.

   Key policy choices to lock down:

   •  required  +  defaultValue : either forbid this combo, or allow it and treat default as satisfying missing input.
   • Type parsing:  defaultValue  is annotation text, so processor/reflection path must parse/validate it against    param type (fail fast on mismatch).
   • Path parity: compile-time generated path and runtime reflection fallback must apply identical defaulting    semantics.

   So your instinct is right: no lower-level CLI blocker; it’s a Java API/runtime behavior decision.
