Here's the detailed analysis:

## What `SchemaGeneratorTest` tests

It verifies that `SchemaGenerator` — a **compile-time** utility — correctly maps `javax.lang.model` types (the type system available inside annotation processors) to JSON Schema expressed as Java source code literals (`Map.of(...)` / `List.of(...)` strings).

## Testing mechanism: compilation-testing

The test can't just call `SchemaGenerator` directly because its API requires `TypeMirror`, `Types`, and `Elements` — objects that only exist during `javac` annotation processing. So it uses a **compilation-testing** pattern:

1. **`InMemorySource`** — wraps a Java source `String` as a `SimpleJavaFileObject`, so `javac` can compile it without writing source files to disk.

2. **`SchemaCapturingProcessor`** — a custom annotation processor (`@SupportedAnnotationTypes("*")`) that runs during compilation. Inside `process()`, it:
   - Scans each root element (class/record/enum/interface) for methods whose name starts with `schemaTarget` — then calls `generator.generateSchemaSource()` on that method's **return type**, recording `"methodName=schemaOutput"` in a static list.
   - For methods named `parametersTarget`, calls `generator.generateParametersSchemaSource()` on the method's parameter list.
   - For root types named `TestRecord*`, `TestEnum*`, or `TestSealed*`, generates schema for the type itself (records, enums, sealed interfaces).

3. **`compileAndCapture()`** / **`compileAndCaptureParams()`** — helper methods that:
   - Create an in-memory compilation unit from the source string
   - Invoke `JavaCompiler.getTask()` with a `StandardJavaFileManager` configured to write `.class` output to `target/test-schema-classes/`
   - Register `SchemaCapturingProcessor`
   - Call `task.call()` to trigger compilation → annotation processing → schema capture
   - Return the captured schema strings

## What the 24 tests assert

Each test provides a tiny Java source snippet with a strategically typed method, compiles it, and asserts that `SchemaGenerator` produces the exact expected `Map.of(...)` string:

| Category | Tests | Example assertion |
|---|---|---|
| **Primitives & boxed** | `int`, `long`, `double`, `float`, `boolean` + `Integer`, `Boolean` | `int` → `Map.of("type", "integer")` |
| **String** | `stringType` | → `Map.of("type", "string")` |
| **Arrays** | `stringArrayType` | `String[]` → `Map.of("type", "array", "items", Map.of("type", "string"))` |
| **Collections** | `listOfStringType` | `List<String>` → same as array of string |
| **Maps** | `mapStringString`, `mapStringObject`, `mapStringBoolean`, `mapStringLong` | `Map<String, V>` → `Map.of("type", "object", "additionalProperties", ...)` with typed values, or just `"object"` for `Object` values |

The only downside is a **slightly different user experience** for the one-record-parameter case.

**With the shortcut**, a tool like:

record SearchArgs(String query, int limit) {}
