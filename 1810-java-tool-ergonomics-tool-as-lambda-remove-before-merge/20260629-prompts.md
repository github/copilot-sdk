### 14:21 Prompt

### 14:22 Response

Yes — mostly a core language/runtime difference.

C# gives you:

• a universal  Delegate  abstraction
• rich reflection over delegate signatures
• natural sync/async and multi-arity expression through  Func<> / Action<> 

Java gives you:

• individual SAM types, not one universal callable type
• no built-in  Function3 ,  Function4 , etc.
• weaker lambda introspection for metadata/signature recovery

Java usually has to choose between:

• explicit overload/SAM families, or
• a lower-level reflective/invocation-based API.

/**
 * Runtime parameter metadata for lambda-defined tools.
 * Mirrors the fields of @CopilotToolParam.

public final class Param {

    private final String value;        // description
    private final String name;         // parameter name override
    private final boolean required;    // required flag
    private final String defaultValue; // default value as string

    private Param(String value, String name, boolean required, String defaultValue) {
        this.value = value == null ? "" : value;
        this.name = name == null ? "" : name;
        this.required = required;
        this.defaultValue = defaultValue == null ? "" : defaultValue;
    }

    /** Minimal fluent entrypoint (required=true, defaultValue=""). */
    public static Param of(String name, String value) {
        return new Param(value, name, true, "");
    }

    /** Full factory for parity with annotation fields. */
    public static Param of(String name, String value, boolean required, String defaultValue) {
        return new Param(value, name, required, defaultValue);
    }

    // Fluent modifiers
    public Param name(String name) {
        return new Param(this.value, name, this.required, this.defaultValue);
    }

    public Param value(String value) {
        return new Param(value, this.name, this.required, this.defaultValue);
    }

    public Param required(boolean required) {
        return new Param(this.value, this.name, required, this.defaultValue);
    }

    public Param defaultValue(String defaultValue) {
        return new Param(this.value, this.name, this.required, defaultValue);
    }

    // Accessors intentionally match annotation element names
    public String value() { return value; }
    public String name() { return name; }
    public boolean required() { return required; }
    public String defaultValue() { return defaultValue; }

    @Override

Looking back at the issue spec (#1831), it says:

If a `@CopilotTool` method declares a parameter of type `com.github.copilot.rpc.ToolInvocation`, the annotation processor must:
• exclude that parameter from generated JSON Schema
• pass the current `invocation` object directly when generating the method call

It does not specify position requirements or constraints. It doesn't say "must be last" nor does it say "can be anywhere."

So: the spec is silent on position. The implementation happens to support any position, but there's no explicit requirement either way.
