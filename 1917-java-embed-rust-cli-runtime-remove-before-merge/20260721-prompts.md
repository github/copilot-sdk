5. The Copilot SDK for Java currently has no embedded runtime. It depends on an externally provided runtime process (see epic [#1917](https://github.com/github/copilot-sdk/issues/1917)). The ongoing Rust port of the `copilot-agent-runtime` repository is reaching the point where the runtime can be consumed as a native shared library without requiring a Node.js process, making it practical to embed the runtime directly in the SDK JAR.
6. 
7. ### The runtime artifact
8. 
9. The artifact to be embedded is `runtime.node`, a Rust [`cdylib`](#references) produced by the `src/runtime` crate in `github/copilot-agent-runtime` using the [napi-rs](#references) build toolchain. Despite the `.node` file extension (a naming convention of napi-rs), this is an ordinary platform-specific shared library (`.so` on Linux, `.dylib` on macOS, `.dll` on Windows). It exposes two front doors built over the same internal engine:
10. 
11. - **[napi](#references) front door** — loaded by a Node.js process as a native addon (current CLI path).
12. - **[C ABI](#references) front door** — a fixed set of approximately 12 `extern "C"` lifecycle and transport entry points (`copilot_runtime_server_create`, `copilot_runtime_connection_open`, etc.) that any language can call in-process via [FFI](#references) ([JNA](#references) for Java, Python/cffi, C#/`DllImport`, Go/purego) **without a Node.js process**. All API methods travel as JSON-RPC data through this fixed transport; the export list never changes as the method set grows.
13. 
14. The `cli-native.node` addon — a separate, smaller artifact that provides ICU4X text segmentation, Win32 API wrappers, and terminal UI helpers — is a CLI-only artifact used by the Ink/React terminal interface. It is **not needed** by the Java SDK.
15. 
16. ### Note on the active Rust migration
17. 
18. As of 2026-07, the `runtime.node` binary is being built up iteratively as TypeScript runtime code is ported into it. It is **not** being reduced; it is growing with each port PR. The `embedded_host.rs` module in the runtime currently spawns a short-lived child process to service method bodies not yet ported to Rust. This internal Node.js dependency shrinks with each port PR and is expected to disappear entirely when the migration completes. The C ABI surface and loading mechanism described in this ADR are stable regardless of migration progress.
19. 
20. ### Platform dimensions
21. 
22. The runtime must be built for each unique combination of OS, CPU architecture, and (on Linux) C runtime variant. The build system in `github/copilot-agent-runtime` produces eight Rust target triples:
23. 
24. | Platform label | Rust triple | Constraint |
25. |---------------|-------------|------------|
26. | `linux-x64` | `x86_64-unknown-linux-gnu` | [glibc](#references) ≥ 2.28 (Debian 10+, Ubuntu 20.04+, RHEL 8+) |
27. | `linux-arm64` | `aarch64-unknown-linux-gnu` | glibc ≥ 2.28 |
28. | `linuxmusl-x64` | `x86_64-unknown-linux-musl` | dynamically links [musl libc](#references) (Alpine Linux) |
29. | `linuxmusl-arm64` | `aarch64-unknown-linux-musl` | dynamically links musl libc |
30. | `darwin-x64` | `x86_64-apple-darwin` | macOS, Intel |
31. | `darwin-arm64` | `aarch64-apple-darwin` | macOS, Apple Silicon |
32. | `win32-x64` | `x86_64-pc-windows-msvc` | [MSVC CRT](#references) statically linked (`+crt-static`) |
33. | `win32-arm64` | `aarch64-pc-windows-msvc` | MSVC CRT statically linked (`+crt-static`) |
34. 
35. The GNU/Linux glibc minimum of 2.28 is enforced at build time via a Microsoft/vscode-linux-build-agent sysroot and verified post-build by `script/linux/verify-glibc-requirements.sh`. The musl binaries are **not** fully statically linked; they dynamically link musl libc (`-C target-feature=-crt-static` is explicitly set at build time).
36. 
37. The **common case** (Windows × 2 + macOS × 2 + GNU/Linux × 2) requires **6 binaries**. Supporting Alpine Linux adds 2 more musl binaries for a total of **8**.
38. 
39. ### Platform selection is 100% deterministic
40. 
41. The correct binary can be selected at runtime without any heuristics, using only standard Java and OS APIs:
42. 
43. 1. **OS**: `System.getProperty("os.name")` — distinguishes Windows, macOS, and Linux unambiguously.
44. 2. **Architecture**: `System.getProperty("os.arch")` — `"amd64"` and `"x86_64"` both map to `x64`; `"aarch64"` and `"arm64"` both map to `arm64`.
45. 3. **Linux libc variant**: Read the first 2 KB of `/proc/self/exe` and parse the [ELF](#references) PT_INTERP segment (the dynamic linker path). If the interpreter path contains `/ld-musl-` → musl; if it contains `/ld-linux-` → glibc. This requires no subprocess, no PATH lookup, and works inside containers. This is the same approach used by the `detect-libc` npm package (its primary, most reliable detection method).

66. ## Considered Options
67. 
68. ### Option 1: Monolithic JAR — all platform binaries in one artifact
69. 
70. All 6 (or 8) `runtime.node` binaries are bundled inside the single `copilot-sdk-java` artifact. At runtime the SDK extracts and loads the one matching the current platform; the remaining 5–7 are carried silently.
71. 
72. **Advantages:**
73. - Single `<dependency>` in `pom.xml`; zero extra configuration for users.
74. - Familiar pattern: [ONNX Runtime](#references) (`onnxruntime-1.21.0.jar`, **130 MB**, all platforms) demonstrates this is an accepted norm in the Java ML ecosystem.
75. 
76. **Drawbacks:**
77. - Every user downloads every platform regardless of their target. A developer on Apple Silicon downloads 105+ MB of Linux and Windows binaries they will never use.
78. - Build tooling (thin Docker layers, incremental CI caches, artifact registries) penalises large JARs. A single 132–180 MB JAR invalidates the entire cache whenever any platform's binary changes.
79. - Maven's dependency resolution has no mechanism to supply platform-appropriate variants automatically; platform selection must happen entirely at runtime inside the JAR.
80. - Conflicts with the principle that Maven artifacts should be reproducible and minimal.
81. 
82. ### Option 2: Per-platform classifier JARs ([DJL](#references) style)
83. 
84. A small, pure-Java coordination artifact (`copilot-sdk-java`, ~1.5 MB) is published alongside separate per-platform native artifacts differentiated by Maven classifier:
85. 
86. ```
87. com.github:copilot-sdk-java-runtime:VERSION:linux-x64
88. com.github:copilot-sdk-java-runtime:VERSION:linux-arm64
89. com.github:copilot-sdk-java-runtime:VERSION:linuxmusl-x64
90. com.github:copilot-sdk-java-runtime:VERSION:linuxmusl-arm64
91. com.github:copilot-sdk-java-runtime:VERSION:darwin-x64
92. com.github:copilot-sdk-java-runtime:VERSION:darwin-arm64
93. com.github:copilot-sdk-java-runtime:VERSION:win32-x64
94. com.github:copilot-sdk-java-runtime:VERSION:win32-arm64
95. ```
96. 
97. Each classifier JAR contains only the `runtime.node` binary for that platform (~19–26 MB compressed) plus a small `.properties` metadata file. The coordination artifact selects and loads the matching native at startup.
98. 
99. This is the same pattern used by DJL's PyTorch native artifacts (`pytorch-native-cpu-2.5.1-linux-x86_64.jar`, `pytorch-native-cpu-2.5.1-osx-aarch64.jar`, etc.), Netty's `netty-tcnative-boringssl-static` per-platform JARs, and others.

154. ## Binding technology: JNA over Panama FFM
155. 
156. A secondary decision within the scope of this ADR is *how* the coordination artifact calls the C ABI entry points once the correct `runtime.node` binary has been loaded. Two candidates were considered: [JNA](#references) and the [Foreign Function & Memory API](#references) (FFM, the product of [Project Panama](#references), final since Java 22 via [JEP 454](#references)).
157. 
158. **Chosen: JNA.** FFM was considered and deliberately deferred, for the following reasons:
159. 
160. 1. **Java baseline.** The SDK supports Java 17, where FFM does not exist (it finalized in Java 22). A JNA-based binding is therefore required regardless; adopting FFM today would mean maintaining two parallel binding implementations, not replacing one with the other.
161. 
162. 2. **Consumer-side configuration burden.** FFM downcalls and upcalls are restricted operations under the JDK's integrity-by-default direction ([JEP 472](#references)). An FFM-based SDK would require every consumer to grant native access explicitly — `--enable-native-access=<module>` (or `ALL-UNNAMED` for classpath applications) on the launcher, or an `Enable-Native-Access` manifest attribute. JNA requires no consumer-side configuration today. For an SDK, this flag becomes every downstream application's problem and a predictable source of support issues. (JNA is on the same enforcement trajectory eventually, as it uses JNI internally; this consideration buys time, not immunity.)
163. 
164. 3. **No realizable performance benefit.** FFM's principal advantage over JNA is the elimination of per-call reflective marshalling overhead. The C ABI surface here is a fixed set of ~12 entry points carrying JSON-RPC strings; JSON serialization/deserialization cost dominates the call path, and call frequency is bounded by agent-interaction rates rather than tight loops. The latency difference between JNA and FFM is expected to be unmeasurable in end-to-end SDK usage. This calculus would change only if the transport moved to a high-frequency or shared-memory framing model.
165. 
166. 4. **Upcall lifetime complexity.** The transport is bidirectional: the runtime delivers JSON-RPC responses and server-initiated requests back into Java from native threads. JNA's `Callback` mechanism handles foreign-thread attachment with well-established semantics. FFM upcall stubs require explicit `Arena` lifetime management, where a stub whose arena is closed while the Rust side still holds the function pointer results in a JVM crash. This shifts lifetime reasoning that JNA encapsulates onto the binding layer.
167. 
168. 5. **GraalVM native-image maturity.** JNA's behavior under GraalVM native-image is well established with mature reachability metadata. FFM support in native-image (particularly for upcalls) is newer and varies by GraalVM release. Plausible SDK consumers (e.g., Quarkus/Micronaut-based CLI tools) compile to native images, so this is a compatibility surface the SDK should not destabilize without verification.
169. 
170. 6. **FFM's safety advantages do not apply to this ABI shape.** FFM's `MemorySegment` bounds and lifetime checking pays off when Java code performs structural manipulation of native memory. This surface passes strings through a fixed transport; there is little structural memory work to make safe.
171. 
172. ### Preserving the FFM migration path
173. 
174. FFM is regarded as the likely eventual binding technology: the JEP 472 endgame applies enforcement pressure to JNA as well, and a ~12-function stable C ABI makes a future migration inexpensive. To keep that path open at low cost:
175. 
176. - The binding layer is abstracted behind a small internal interface (native load + downcall + upcall registration), so that an FFM implementation can be introduced later — for example, as a multi-release JAR selecting FFM on Java 22+ — without changes to the transport or API layers.
177. - The decision should be revisited when (a) the SDK's minimum Java baseline moves past 17, or (b) JDK releases begin enforcing `--illegal-native-access=deny` by default, whichever comes first.

Hence the decision: per-platform classifier JARs, DJL-style — a ~1.5 MB pure-Java coordination artifact plus one ~20–26 MB platform JAR selected via os-maven-plugin or Gradle variant resolution, with musl detection done by parsing the ELF PT_INTERP segment of `/proc/self/exe`, the same approach as the detect-libc npm package.
164. 
165. So your Epic text's third paragraph — "the download size of the Copilot SDK Java jar will still be acceptably small" — is true only under the classifier-JAR strategy, not under naive embedding. 170. 
171. ### 16:05 Prompt
172. 
173. Aside from requiring Java 25 where we formerly support 17 or 25, are there any other downsides to using Panama instead of JNA?
174. 
175. ### 16:10 Response
176. 
177. Yes, several — and a couple of them are more consequential for an SDK than for an application, because every cost you take on is inherited by all downstream consumers.
178. 
179. **The `--enable-native-access` requirement is the big one.** JNA today loads and calls native code with at most a warning. Under FFM plus the JEP 472 integrity-by-default trajectory, restricted native operations require the consumer to grant native access — `--enable-native-access=com.github.copilot.sdk` (or `ALL-UNNAMED` for classpath users) on the launcher, or an `Enable-Native-Access` manifest attribute for executable JARs. That's friction you're pushing onto every Java app, Spring Boot service, and CI harness that adopts the SDK, and it's the kind of thing that generates support issues ("why does my app print/throw this?"). The irony is that JNA itself sits on the same enforcement path — it uses JNI internally, and `--illegal-native-access` will eventually bite it too — so this is a downside of *now vs. later*, not *ever vs. never*. But "later" may buy you a year or two of zero-config UX.
180. 
181. **GraalVM native-image support is weaker.** JNA on native-image is a known quantity with established reachability metadata. FFM support in native-image landed later and has been maturing incrementally — downcalls arrived first, upcalls and broader coverage have been filling in, and behavior varies by GraalVM release. If any meaningful slice of Copilot SDK consumers compile to native images (Quarkus/Micronaut CLI tools are plausible consumers), you'd want to verify the current state on your supported GraalVM versions before switching. This one is checkable and changing fast, so treat it as a verification item, not a permanent objection.
182. 
183. **Callback/upcall ergonomics and threading.** The C ABI transport is presumably bidirectional — the runtime needs to deliver JSON-RPC responses and server-initiated requests back to Java, which means native-to-Java calls. JNA's `Callback` handles foreign-thread attachment with well-worn semantics. FFM upcall stubs work from native threads too, but you're managing `Arena` lifetimes for the stubs yourself — an upcall stub whose arena closes while Rust still holds the function pointer is a JVM crash, and the Rust side's threading model (which thread invokes the callback, during or after which calls) becomes something you must reason about explicitly rather than something the binding layer papers over.
184. 
185. **You lose JNA's crash padding, and FFM's safety wins don't apply here.** FFM's safety story (bounds-checked `MemorySegment`, lifetime scopes) pays off when Java manipulates native memory structurally. This surface is ~12 functions passing JSON strings — there's little structural memory work to make safe. Meanwhile JNA offers protected mode as a best-effort net around native faults. Neither matters enormously, but the net safety delta for *this particular ABI shape* is close to zero.
186. 
187. **The performance argument is weaker than it looks, for the same reason.** JNA's overhead is per-call marshalling; FFM largely eliminates it. But your payloads are JSON-RPC strings whose serialization/deserialization cost dwarfs the call overhead, and call frequency is bounded by agent-interaction rates, not tight loops. You'd be taking on all the above costs for a latency win that's probably unmeasurable in end-to-end SDK usage. (If the transport ever moves to something like shared-memory frames or high-frequency streaming chunks, this calculus changes.)
188. 
189. **Finally, the boring ones:** it's a departure from what ADR-007 currently specifies, so it's an ADR amendment plus re-validation across all eight platform targets; hand-written FFM bindings are genuinely fine for 12 functions but are one more thing to keep in sync as the entry-point set evolves (the ADR's whole design bet is that it *won't* evolve, which helps); and team/reviewer familiarity with JNA is likely higher today.
190. 
191. The honest synthesis for the commenter: Panama is the right *eventual* answer — the JEP 472 endgame pushes everyone there, and a 12-function stable C ABI is an easy migration. But on a 17-or-25 dual baseline it's not available at all on 17, and even on a 25-only baseline the immediate trade is "consumer launcher flags + native-image verification + upcall lifetime management" purchased with a performance benefit that JSON transport mostly nullifies. A reasonable middle path is to structure the loading/binding layer behind a small internal interface so an FFM implementation can ship later as a multi-release JAR or optional module without touching the transport logic.
