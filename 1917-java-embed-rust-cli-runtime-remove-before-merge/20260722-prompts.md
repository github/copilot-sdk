## Why it works

npm already handles integrity verification via `package-lock.json` (SHA-512 in the `integrity` field). The Java build already has precedent for running `npm ci` during `generate-test-resources`. The key insight: if you list the platform packages as **regular** `dependencies` (not `optionalDependencies`), npm installs them regardless of the host OS — it's only `optionalDependencies` that get platform-skipped.

## Proposed architecture

```
java/
├── pom.xml                          (parent reactor POM, packaging=pom)
├── sdk/
│   └── pom.xml                      (existing SDK module, unchanged)
├── native-deps/
│   ├── pom.xml                      (module that runs npm ci)
│   └── package.json                 (lists all 8 platform packages)
│   └── package-lock.json            (integrity hashes, committed)
├── native-linux-x64/
│   └── pom.xml                      (classifier JAR: embeds copilot + runtime.node)
├── native-darwin-arm64/
│   └── pom.xml
│   ...
```

1. `native-deps` — `generate-resources` runs `npm ci`, placing binaries at known paths

```json
    "@github/copilot-linux-x64": "1.0.69-2",
    "@github/copilot-linux-arm64": "1.0.69-2",

    "@github/copilot-darwin-x64": "1.0.69-2",
    "@github/copilot-darwin-arm64": "1.0.69-2",
```

## Simpler alternative: keep single module

You don't strictly need a multi-module reactor. The existing single-module POM can do this:

1. Add an `exec-maven-plugin` execution in `generate-resources` (before `compile`) that runs `npm ci` in a `native-deps/` directory within java
2. Use `maven-jar-plugin` additional executions (or `maven-assembly-plugin`) to produce classifier JARs from those extracted paths
3. The main `mvn clean verify` still works without native binaries — the download execution is gated behind a profile (e.g., `-Pbundle-native`)

## Trade-offs vs the other SDKs

| Concern | This approach | Rust approach |
|---------|---------------|---------------|
| Integrity verification | npm lockfile SHA-512 (same trust source) | Manual SHA-512 from lockfile |
| Registry configurability | npm's `.npmrc` — supports private registries natively | Hardcoded URL with env override |
| Build dependency | Node.js + npm (already required for test harness) | None (Rust's build.rs has no external deps) |
| Offline builds | `npm ci --offline` works if cache is warm | Custom cache dir logic |

Since the Java E2E tests already require Node.js (for the replay proxy), the Node.js build dependency is **not** a new requirement — it's already there. This makes the package.json approach the path of least resistance.

## Supply-chain security note

This approach inherits npm's integrity verification automatically. The committed `package-lock.json` contains `integrity: "sha512-..."` for each package — `npm ci` verifies these on every install. This is strictly better than what Go and Python currently do (no verification) and equivalent to what Rust does (manual SHA-512 check against the same lockfile).

### Recommendation

**Single-module with profile** is the pragmatic choice because:

1. It avoids a breaking GAV change
2. It mirrors how .NET and Rust handle this (single build unit, conditional native bundling)
3. The existing POM already uses `exec-maven-plugin` + `npm ci` for the test harness — adding another execution for native deps is a natural extension
4. Classifier JARs attached to the same artifact deploy atomically — one staging repo, one signature, one release
5. The multi-module approach can be introduced later if the single POM becomes genuinely unmanageable — but 8 `maven-jar-plugin` executions (each ~15 lines) is modest

The escape hatch is clean: if the POM grows unwieldy, you extract the native packaging into a child module without changing the SDK's published GAV.

**Yes, you can apply the same multi-release JAR pattern to the callback/FFI side.** The optimization opportunity is significant:

### JNA (Java 17 baseline) — what the spike proved works
- JNA creates a **new short-lived Java platform thread per callback invocation** (Thread-0, Thread-1, Thread-2...)
- Requires `QueueInputStream` because `PipedInputStream` thread-affinity checks fail
- JNA copies native buffers via `Pointer.getByteArray()`
- Requires `--enable-native-access=ALL-UNNAMED` (JNA uses restricted methods)

### FFM (Java 25 multi-release overlay) — what it could improve
1. **No thread creation overhead** — FFM upcall stubs (`Linker.nativeLinker().upcallStub()`) execute directly on the calling native thread. No new Java thread per callback. The Rust thread calls the function pointer and FFM handles it in-place.
2. **Zero-copy buffer access** — `MemorySegment.ofAddress(ptr, len)` wraps the native buffer without copying. You can call `MemorySegment.asByteBuffer()` or `.toArray()` only when needed.
3. **No JNA dependency at runtime on JDK 25+** — FFM is built into the JDK. The `jna` JAR is still needed at compile time for the baseline path, but JDK 25 consumers get pure-JDK native access.
4. **PipedInputStream might work again** — since FFM runs callbacks on the same native thread (no new threads), the `writeSide.isAlive()` check wouldn't fail. But `QueueInputStream` is still simpler and thread-model-agnostic.

Here are the three .NET connection types. They're not an enum — they're a sealed class hierarchy rooted at `RuntimeConnection`:

**1. `StdioRuntimeConnection`** (`RuntimeConnection.ForStdio()`)
- Spawns the CLI as a **child process**, communicates over **stdin/stdout** pipes
- This is the **default** if no `Connection` is set on options
- Extends `ChildProcessRuntimeConnection` (has `Path`, `Args`, `Environment`)

**2. `TcpRuntimeConnection`** (`RuntimeConnection.ForTcp(port, connectionToken)`)
- Spawns the CLI as a **child process**, but the child listens on a **TCP socket** instead of stdio
- SDK connects to it via TCP
- Also extends `ChildProcessRuntimeConnection`
- Has additional `Port` and `ConnectionToken` properties

**3. `InProcessRuntimeConnection`** (`RuntimeConnection.ForInProcess()`)
- **No child process** — loads the native `runtime.node` library in-process via FFI (C ABI)
- Marked `[Experimental]`
- Extends `RuntimeConnection` directly (not `ChildProcessRuntimeConnection` — no Path/Args needed)

So .NET has **two subprocess variants** (stdio vs TCP) plus the **in-process FFI variant**. The Java SDK currently has the equivalent of both subprocess variants (the `useStdio` flag and `cliUrl` for TCP), and we're adding the in-process one.

Given the existing API, there are really three options. Let me lay out the tradeoffs:

**Option A: Another boolean — `setUseInProcess(true)`**
Follows the `setUseStdio()` pattern. But now you have two booleans that interact: what does `setUseStdio(false).setUseInProcess(true)` mean? What about `setUseStdio(true).setUseInProcess(true)`? Validation becomes a minefield. **Reject.**

**Option B: Enum — `setTransport(Transport.IN_PROCESS)`**
Clean, but `Transport.CLI_STDIO` vs `Transport.CLI_TCP` doesn't capture the associated config (cli path, port, connection token). You'd still need the existing fields, plus validation that the right fields are set for the chosen enum value. It's half a redesign that doesn't stand on its own.

**Option C: Add `setConnection(RuntimeConnection)` alongside existing fields**
This is what .NET did. Add one new field with a typed hierarchy:

```java
// New in-process path:
var options = new CopilotClientOptions()
    .setConnection(RuntimeConnection.forInProcess());

// Existing code — unchanged, keeps working:
var options = new CopilotClientOptions()
    .setCliPath("/usr/local/bin/copilot");
```

Internally: if `connection` is non-null, use it. If null, infer from the existing `useStdio`/`cliUrl`/`cliPath` fields exactly as today. Zero breakage.

**Option C is the answer.** It's least disruptive because:
- Existing code doesn't change at all — no deprecation needed immediately
- No boolean interaction problems
- Each `RuntimeConnection` subclass carries its own config (in-process needs nothing; URI carries url + token)
- Matches the .NET precedent, so cross-SDK documentation is consistent
- Future transports just add a new factory method

The only design decision is whether to eventually deprecate `setUseStdio()`/`setCliUrl()` in favor of `setConnection(RuntimeConnection.forStdio(...))` — but that's a separate, future step. For now, both paths coexist.

```java
    permits StdioRuntimeConnection, TcpRuntimeConnection,

    RuntimeConnection() {} // package-private constructor

    public static StdioRuntimeConnection forStdio() { return new StdioRuntimeConnection(); }
    public static StdioRuntimeConnection forStdio(String path) { return new StdioRuntimeConnection().setPath(path); }
    public static TcpRuntimeConnection forTcp() { return new TcpRuntimeConnection(); }
    public static UriRuntimeConnection forUri(String url) { return new UriRuntimeConnection(url); }
    public static InProcessRuntimeConnection forInProcess() { return new InProcessRuntimeConnection(); }
}

// --- Stdio (default): spawn CLI, communicate over stdin/stdout ---
public final class StdioRuntimeConnection extends RuntimeConnection {
    private String path;           // null = bundled runtime
    private List<String> args;

    public String getPath() { return path; }
    public StdioRuntimeConnection setPath(String path) { this.path = path; return this; }
    public List<String> getArgs() { return args; }
    public StdioRuntimeConnection setArgs(List<String> args) { this.args = args; return this; }
}

// --- TCP: spawn CLI, connect via TCP socket ---
public final class TcpRuntimeConnection extends RuntimeConnection {
    private String path;           // null = bundled runtime
    private int port;              // 0 = auto-allocate
    private String connectionToken;
    private List<String> args;
    // fluent getters/setters...
}

// --- URI: connect to already-running external CLI ---
public final class UriRuntimeConnection extends RuntimeConnection {
    private final String url;      // required
    private String connectionToken;

    UriRuntimeConnection(String url) { this.url = Objects.requireNonNull(url); }
    // fluent getters/setters...
}

// --- In-Process: load native library via JNA, FFI over C ABI ---
public final class InProcessRuntimeConnection extends RuntimeConnection {
    // No fields needed — uses bundled native library
}
```

- **Sealed class** (Java 17+) prevents third-party subclasses — `CopilotClient` can exhaustively `switch` on the type

- **Validation**: if both `connection` and legacy fields are set, throw `IllegalArgumentException` at construction time
- **.NET parity**: 1:1 mapping with `RuntimeConnection.ForStdio/ForTcp/ForUri/ForInProcess`

Here's the analysis:

---

**Option A: New `FfiRuntimeHost` (parallel class)**

Pros:
- **Single Responsibility.** `CliServerManager` is 100% about subprocess lifecycle: `ProcessBuilder`, command-line args, `waitForPortAnnouncement`, stderr pumping, `Process` cleanup. `FfiRuntimeHost` is 100% about FFI lifecycle: `host_start`, `connection_open`, `QueueInputStream`, `connection_write`, `host_shutdown`. Zero overlap in mechanics.
- **Matches .NET exactly.** .NET has `FfiRuntimeHost` as a completely separate class. Same name, same pattern — cross-SDK consistency.
- **No risk to existing path.** `CliServerManager` is untouched. No accidental breakage of subprocess transport when adding FFI code.
- **Cleaner `CopilotClient` dispatch.** The `startCoreBody()` method already branches on connection type. Adding a third branch (`if connection is InProcessRuntimeConnection → use FfiRuntimeHost`) is natural:

```java
if (connection instanceof InProcessRuntimeConnection) {
    ffiHost = new FfiRuntimeHost(...);
    ffiHost.start();
    rpc = JsonRpcClient.fromStreams(ffiHost.getReceiveStream(), ffiHost.getSendStream());
} else if (optionsHost != null) {
    rpc = serverManager.connectToServer(null, optionsHost, optionsPort);
} else {
    // existing subprocess path unchanged
}
```

Cons:
- **Some shared logic is duplicated.** Building `--embedded-host`, `--no-auto-update`, env var setup for auth tokens, telemetry, etc. Both classes need to construct arg arrays from `CopilotClientOptions`. (Mitigated: extract a shared static helper for arg/env building if needed — but that's a small refactor, not an architectural concern.)

**Option B: Extend `CliServerManager`**

Pros:
- One class to maintain for all server lifecycle.
- Shared arg/env building code stays in one place.

Cons:
- **Violates SRP.** The class would mix `ProcessBuilder`/subprocess concerns with JNA/FFI concerns. Conditionals everywhere: `if (inProcess) { ... } else { ... }`.
- **Name becomes misleading.** "Cli**Server**Manager" doesn't manage a server when running in-process — it manages a native library.
- **Higher change risk.** Every FFI modification touches the same class used by the stable subprocess path.
- **No .NET precedent.** .NET didn't do this — it keeps `FfiRuntimeHost` completely separate from the subprocess path.

---

**Verdict:** Your instinct is right. **New `FfiRuntimeHost`** — parallel class, no shared inheritance with `CliServerManager`. If arg/env building duplication bothers us later, extract a small static helper. But don't couple the two lifecycles.

This is straightforward — no spike needed. Here's why:

`JsonRpcClient` takes a plain `InputStream` + `OutputStream` in its constructor. It already has two factory methods (`fromProcess`, `fromSocket`). We just need a third:

```java
public static JsonRpcClient fromStreams(InputStream in, OutputStream out) {
    return new JsonRpcClient(in, out, null, null);
}
```

Both streams are already solved:

1. **Read side (`InputStream`):** `QueueInputStream` from the 3.4 spike. The JNA `on_outbound` callback pushes `byte[]` into a `BlockingQueue`; `QueueInputStream.read()` drains it. Proven on both JDK 17 and 25.

2. **Write side (`OutputStream`):** A trivial `FfiOutputStream` that wraps `connection_write`:

```java
class FfiOutputStream extends OutputStream {
    private final CopilotRuntimeLibrary lib;
    private final int connectionId;

    @Override
    public void write(byte[] b, int off, int len) {
        // JNA copies the buffer synchronously before returning
        byte[] slice = (off == 0 && len == b.length) ? b : Arrays.copyOfRange(b, off, off + len);
        if (!lib.copilot_runtime_connection_write(connectionId, slice, new NativeSize(len))) {
            throw new IOException("connection_write failed");
        }
    }

    @Override
    public void write(int b) { write(new byte[]{(byte) b}, 0, 1); }
}
```

The only nuance is that `JsonRpcClient` currently stores `Socket socket` and `Process process` for cleanup in `close()`. With `fromStreams(in, out)` both are null — cleanup just closes the streams themselves, which is correct (the `FfiRuntimeHost` owns the native lifecycle separately).

No spike needed. The 3.4 spike already proved the hard part (callback → `QueueInputStream`). The write side is a trivial `OutputStream` wrapper. The `JsonRpcClient` constructor already accepts raw streams.
