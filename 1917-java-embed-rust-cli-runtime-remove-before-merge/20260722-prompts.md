## Why it works

npm already handles integrity verification via `package-lock.json` (SHA-512 in the `integrity` field). The Java build already has precedent for running `npm ci` during `generate-test-resources`. The key insight: if you list the platform packages as **regular** `dependencies` (not `optionalDependencies`), npm installs them regardless of the host OS — it's only `optionalDependencies` that get platform-skipped.

## Proposed architecture

```
java/
├── pom.xml                          (parent reactor POM, packaging=pom)
├── sdk/
│   └── pom.xml                      (existing SDK module, unchanged)
├── native-deps/
│   ├── pom.xml                      (module that runs npm ci)
│   └── package.json                 (lists all 8 platform packages)
│   └── package-lock.json            (integrity hashes, committed)
├── native-linux-x64/
│   └── pom.xml                      (classifier JAR: embeds copilot + runtime.node)
├── native-darwin-arm64/
│   └── pom.xml
│   ...
```

1. `native-deps` — `generate-resources` runs `npm ci`, placing binaries at known paths

```json
    "@github/copilot-linux-x64": "1.0.69-2",
    "@github/copilot-linux-arm64": "1.0.69-2",

    "@github/copilot-darwin-x64": "1.0.69-2",
    "@github/copilot-darwin-arm64": "1.0.69-2",
```

## Simpler alternative: keep single module

You don't strictly need a multi-module reactor. The existing single-module POM can do this:

1. Add an `exec-maven-plugin` execution in `generate-resources` (before `compile`) that runs `npm ci` in a `native-deps/` directory within java
2. Use `maven-jar-plugin` additional executions (or `maven-assembly-plugin`) to produce classifier JARs from those extracted paths
3. The main `mvn clean verify` still works without native binaries — the download execution is gated behind a profile (e.g., `-Pbundle-native`)

## Trade-offs vs the other SDKs

| Concern | This approach | Rust approach |
|---------|---------------|---------------|
| Integrity verification | npm lockfile SHA-512 (same trust source) | Manual SHA-512 from lockfile |
| Registry configurability | npm's `.npmrc` — supports private registries natively | Hardcoded URL with env override |
| Build dependency | Node.js + npm (already required for test harness) | None (Rust's build.rs has no external deps) |
| Offline builds | `npm ci --offline` works if cache is warm | Custom cache dir logic |

Since the Java E2E tests already require Node.js (for the replay proxy), the Node.js build dependency is **not** a new requirement — it's already there. This makes the package.json approach the path of least resistance.

## Supply-chain security note

This approach inherits npm's integrity verification automatically. The committed `package-lock.json` contains `integrity: "sha512-..."` for each package — `npm ci` verifies these on every install. This is strictly better than what Go and Python currently do (no verification) and equivalent to what Rust does (manual SHA-512 check against the same lockfile).

### Recommendation

**Single-module with profile** is the pragmatic choice because:

1. It avoids a breaking GAV change
2. It mirrors how .NET and Rust handle this (single build unit, conditional native bundling)
3. The existing POM already uses `exec-maven-plugin` + `npm ci` for the test harness — adding another execution for native deps is a natural extension
4. Classifier JARs attached to the same artifact deploy atomically — one staging repo, one signature, one release
5. The multi-module approach can be introduced later if the single POM becomes genuinely unmanageable — but 8 `maven-jar-plugin` executions (each ~15 lines) is modest

The escape hatch is clean: if the POM grows unwieldy, you extract the native packaging into a child module without changing the SDK's published GAV.

### The key Maven mechanics

- `maven-jar-plugin` execution with `<classifier>X</classifier>` → calls `project.getArtifactHandler().addAttachedArtifact()` → the classifier JAR becomes an **attached artifact**
- `central-publishing-maven-plugin` (or `maven-deploy-plugin`) iterates `project.getAttachedArtifacts()` and includes them all in the deployment bundle
- Maven Central's Sonatype validation requires the primary (unclassified) artifact, POM, sources JAR, and javadoc JAR. The classifier JARs are additional files that ride along.

This is exactly what DJL does — just expressed in Gradle's `publications` DSL rather than Maven's plugin execution model.

That's it. At runtime, the SDK's `NativeRuntimeLoader` calls `getResourceAsStream("native/darwin-arm64/runtime.node")`, finds it in the classifier JAR on the classpath, extracts it once, and loads it via JNA.

If the customer later needs to support multiple platforms (e.g., a portable uber-JAR for deployment to both macOS and Linux), they just add more classifier dependencies:

```xml
    <dependency>
        <groupId>com.github</groupId>
        <artifactId>copilot-sdk-java-runtime</artifactId>
        <version>1.0.9-preview.0</version>
        <classifier>linux-x64</classifier>
    </dependency>
```

The classloader merges both JARs' resources on the classpath; `NativeRuntimeLoader` picks the one matching the running OS. No configuration changes, no profiles.

For customers who want **automatic** classifier selection at build time (avoid hardcoding `darwin-arm64`), they can use `os-maven-plugin`:

```xml
<build>
    <extensions>
        <extension>
            <groupId>kr.motd.maven</groupId>
            <artifactId>os-maven-plugin</artifactId>
            <version>1.7.1</version>
        </extension>
    </extensions>
</build>

<dependencies>
    <dependency>
        <groupId>com.github</groupId>
        <artifactId>copilot-sdk-java</artifactId>

    </dependency>
    <dependency>
        <groupId>com.github</groupId>
```

Yes, directly. The hybrid already supports both options simultaneously because the classifier JARs use the `native/<classifier>/runtime.node` resource path convention described in the ADR's "How can we do Option 2 and Option 1" section.

### How it works

**Option 2 (per-platform)** is what the `copilot-native` module produces: 8 classifier JARs, each containing `native/<classifier>/runtime.node`.

**Option 1 (monolithic)** is assembled **at the consumer side** — or optionally by us as a convenience artifact. The consumer uses `maven-assembly-plugin` with `jar-with-dependencies`, which unpacks all dependency JARs and merges their contents. If the consumer declares all 8 classifier dependencies, the uber-JAR ends up containing:

```
native/linux-x64/runtime.node
native/linux-arm64/runtime.node
native/darwin-arm64/runtime.node
...all 8...
```

`NativeRuntimeLoader.loadRuntime()` calls `getResourceAsStream("native/" + detected + "/runtime.node")` — same code, same classpath lookup, works identically whether the resource came from a separate classifier JAR or was merged into an uber-JAR.

### If we want to publish a monolithic convenience artifact ourselves

Add one more module to the reactor:

```
java/
├── pom.xml                          (parent)
├── sdk/pom.xml                      (copilot-sdk-java)
├── copilot-native/pom.xml           (copilot-sdk-java-runtime + classifiers)
├── copilot-native-all/pom.xml       (copilot-sdk-java-runtime-all, monolithic)
```

Where `copilot-native-all/pom.xml`:

```xml
<project>
    <parent>
        <groupId>com.github</groupId>
        <artifactId>copilot-sdk-java-parent</artifactId>
    </parent>
    <artifactId>copilot-sdk-java-runtime-all</artifactId>
    <packaging>jar</packaging>

    <dependencies>
        <!-- All 8 classifier JARs as dependencies -->
        <dependency>
            <groupId>com.github</groupId>
            <artifactId>copilot-sdk-java-runtime</artifactId>
            <version>${project.version}</version>
            <classifier>linux-x64</classifier>

        <dependency>
            <groupId>com.github</groupId>
            <artifactId>copilot-sdk-java-runtime</artifactId>
            <version>${project.version}</version>
            <classifier>darwin-arm64</classifier>
        </dependency>
        <!-- ... 6 more ... -->
    </dependencies>

    <build>
        <plugins>
            <plugin>
                <artifactId>maven-assembly-plugin</artifactId>
                <executions>
                    <execution>
                        <id>assemble-all-platforms</id>
                        <phase>package</phase>
                        <goals><goal>single</goal></goals>

                            <descriptorRefs>
```

### Three consumption tiers for the consumer

| Consumer wants | Dependencies | Download size |
|---|---|---|
| Single platform | `copilot-sdk-java` + `copilot-sdk-java-runtime:classifier=darwin-arm64` | ~21 MB |
| Portable uber-JAR (DIY) | `copilot-sdk-java` + all 8 classifier deps + their own `maven-assembly-plugin` | ~180 MB |
| Portable uber-JAR (pre-built) | `copilot-sdk-java` + `copilot-sdk-java-runtime-all` | ~180 MB |

The third option is the "Option 1 monolithic" from the ADR, just published as a separate artifact rather than being the same artifact. The consumer trades download size for simplicity — one dependency, zero platform decisions.

### Reactor build order

```
copilot-native          →  produces placeholder + 8 classifier JARs
sdk                     →  produces copilot-sdk-java (pure Java)
copilot-native-all      →  depends on all 8 classifiers, assembles monolithic JAR
```

All three modules publish to Central in the same release. Same `central-publishing-maven-plugin`, same GPG key, same workflow.
