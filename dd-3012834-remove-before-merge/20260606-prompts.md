### The Java analogue gap

There's no Java-language equivalent to the sketch. JEP-level discussions have come up sporadically, but nothing has been adopted. The state of the art in Java is roughly what you sketched: an annotation plus Error Prone or a custom processor. Guava's `@Beta` + the BetaApi Error Prone check is the most prominent example, but it's notably less expressive than `@RequiresOptIn`—no propagation model, single binary "is this beta or not" rather than user-defined markers.

If you were designing this from scratch for a Java codebase today, Error Prone gives you most of the building blocks, but you'd be reimplementing the propagation semantics yourself. JSpecify has shown that Java annotations _can_ carry sophisticated semantics with enough tooling investment, but no one has done that work for opt-in specifically.

### Why has this not been done already as described in the sketch?

The sketch _looks_ straightforward, but several factors explain why no widely-adopted Java implementation exists. They're roughly: technical limits of the annotation processor model, ecosystem dynamics, and the existence of "good enough" alternatives.

1. The Tree API problem is bigger than it sounds
------------------------------------------------

The sketch acknowledges this but understates the severity. Annotation processors operating purely through JSR 269 (`javax.lang.model.*`) see **declarations**, not expressions. That means you can check:
*   Field types, method parameter types, return types, thrown exceptions
*   Superclasses and implemented interfaces
*   Type parameters and bounds
*   Annotations on declarations
But you _cannot_ see (through the standard API):
*   `new ExperimentalFoo()` inside a method body
*   `ExperimentalFoo.staticMethod()` calls
*   Local variable types
*   Method references and lambdas referring to experimental types
*   Generic type arguments at call sites
To catch those, you need `com.sun.source.tree` (the Compiler Tree API), which is **not part of the Java SE specification**. It lives in `jdk.compiler` and is a JDK-internal API. Using it ties you to javac specifically—ECJ (Eclipse's compiler) has its own AST, and other compilers may have nothing equivalent. For a library that wants to work across all build environments, this is a real problem.
This is why Error Prone exists as a **javac plugin** rather than a pure annotation processor: it needs the Tree API to do what it does. And Error Prone has effectively occupied the niche the sketch describes.

2. The propagation model needs language support to feel right
-------------------------------------------------------------

The sketch's `-AallowExperimental` flag is binary and global. To get Kotlin-style granularity (per-marker, per-callsite, propagating vs. non-propagating opt-in), you'd need:
*   A way to write `@OptIn(SomeMarker.class)` at arbitrary positions (method, class, statement, expression)
*   Compiler understanding that an annotated scope suppresses warnings for that specific marker
*   Propagation semantics where re-marking with the original annotation is meaningfully different from `@OptIn`
Java annotations can target statements only in limited ways (you can put them on local variable declarations but not on arbitrary expressions). `@SuppressWarnings` works because the compiler has special knowledge of it. Replicating that for user-defined opt-in markers requires either compiler cooperation or, again, a tool like Error Prone that's doing its own AST analysis.
You _can_ approximate it with Error Prone—and some teams have built internal versions of this—but it's not a small project. You're essentially reimplementing Kotlin's opt-in machinery in a tool layer.


3. JSpecify shows what it would take
------------------------------------

JSpecify is the closest recent example of "let's actually standardize a Java annotation with rigorous semantics." It took years of cross-organization work (Google, JetBrains, Oracle, Microsoft, etc.) to converge on nullness annotations, and the tooling story is still being built out. A similar effort for opt-in would need comparable coordination, and nullness arguably had a stronger case (decades of NPEs) than API-stability marking does.

* * *

So the short answer: the sketch describes something real and doable, but the easy version (declaration-only checks via pure JSR 269) catches too little, and the comprehensive version (full AST coverage with propagation) is essentially Error Prone + a significant new check, which is enough work that the modest improvement over `@Beta` hasn't justified anyone building it. Kotlin had a language team and a forcing function; Java has neither for this specific problem.

</global-problem-statement>

What is missing is not one thing, but the full enforcement pipeline between “annotation exists” and “consumer build fails by default.”

The sketch has the idea, but it is missing the concrete pieces below.

**1. A precise enforcement spec (currently implicit)**
You need to define exactly what counts as a forbidden use for your narrowed scope (types + methods only), for example:
1. Referencing an experimental type in declarations (field, param, return, throws, extends, implements, generic bounds).
2. Instantiating an experimental type.
3. Calling an experimental method.
4. Referencing an experimental method via method reference.
5. Accessing members of an experimental type if type-level annotation implies member-level restriction.

Without this spec, checker behavior will be inconsistent and hard to test.

**2. The annotation contract itself**
The sketch does not fully define:
1. Retention policy and targets.
2. Whether type-level annotation implies all methods are experimental.
3. Optional opt-in scope annotation (for source-level opt-in) vs compiler-arg-only opt-in.
4. Diagnostic message format and suppression policy.

**3. AST-level checker implementation (not just declaration model)**
A declaration-only processor is insufficient for your requirement “any use fails.”
You need a javac AST-aware checker (Error Prone check or processor using Trees/TreePathScanner) that inspects expression-level usage:
1. MethodInvocationTree
2. NewClassTree
3. MemberReferenceTree
4. IdentifierTree / MemberSelectTree with symbol resolution

This is the biggest technical gap in the sketch.

**4. Build integration model for real consumers**
This is the operational gap most sketches miss:
1. How the checker is shipped (artifact separation: annotation + checker).
2. How consumer builds enable it automatically or by profile.
3. Default behavior must be fail-closed (experimental off unless explicitly enabled).
4. How Gradle and Maven users pass opt-in (single agreed flag/property).

If consumers do not run the checker, annotation alone does nothing.

**5. Generator changes beyond Javadoc**
Your generator currently emits only Javadoc notes for experimental elements in java.ts, java.ts, and java.ts.
Missing change: emit the actual annotation on those same program elements (types and wrapper methods).

**6. Internal build strategy for this repo**
Once generated code is annotated, your own hand-written Java code that uses generated APIs may fail unless the repo build opts in where intended.
This concern is already hinted in adr-003-sub-module-for-generated-code.md.

**7. Compatibility decision**
You must choose and document:
1. Javac-only enforcement (practical).
2. Behavior for ECJ/IDE compilers where checker may not run.
3. Minimum toolchain versions.

Without this, enforcement may differ between local IDE and CI.

**8. Test matrix**
Missing from the sketch:
1. Negative compile tests (must fail without opt-in).
2. Positive compile tests (must pass with opt-in).
3. Coverage for method calls, type references, inheritance, and method references.
4. Regression tests tied to schema-driven generation.

Given your simplifying assumption, this is very achievable. The key missing piece is: a shipped, AST-level compile checker plus build wiring so it runs for consumers by default, and generator output that places the annotation where you already place apiNote today.
